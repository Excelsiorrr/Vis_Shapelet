# ShapeX 各 Part 复用开发规范

## 0. 文档目的
- 目的: 将 `Part A` 开发过程中踩过的坑、修正过的问题和最终形成的工程做法沉淀为统一规范。
- 适用范围: `Part B / Part C / Part D / Part E` 以及后续新增 Part。
- 使用方式: 每启动一个新 Part，先按本文件走一遍，再开始具体编码。

## 1. 核心原则

### 1.1 先冻结口径，再写代码
- 先更新 PRD，再进入实现。
- 先定义算法口径、接口口径、MVP 边界，再讨论实现细节。
- 不允许一边写代码一边临时改变核心字段语义。

### 1.2 以运行时真实结果为准，不以配置猜测为准
- 配置文件、yaml、历史注释都只能作为参考。
- 真实类别数、真实 shape、真实 split 语义、真实 checkpoint 输出，必须以运行时 dry run 验证结果为准。
- 若配置与运行结果冲突，以运行结果为准，同时回写文档说明差异。

### 1.3 已知偏差必须显式暴露
- 任何已知问题都不能只藏在代码注释里。
- 必须同时体现在以下至少一处，通常应同时体现在两处:
  - PRD 的 `MVP 备注 / 已知问题`
  - API 响应中的 `warnings`
  - 响应中的显式映射字段，例如 `split_mapping`

### 1.4 接口必须按页面加载节奏拆分
- 默认禁止设计单一重型总接口。
- 首屏只返回最小必要信息。
- 高成本计算、列表数据、明细数据全部拆成异步接口。
- 列表类接口必须分页。

### 1.5 适配逻辑必须集中封装
- 数据 shape 对齐、split 映射、checkpoint 元信息推断、默认参数回填等逻辑统一放在 service 层。
- router 只做参数校验和响应编排。
- schema 只定义契约，不承载业务推导。

### 1.6 可复现性是默认要求
- 固定随机种子。
- 明确记录关键参数。
- 明确缓存键。
- 同一输入下应尽量返回一致结果。

### 1.7 MVP 和未来能力必须明确切开
- PRD 可以保留未来能力，但必须标明本版是否落地。
- 对暂不支持的能力，前后端都要有清晰降级策略。

## 2. Part 开发标准流程

## 2.1 第一步: 先改 PRD
- 每个新 Part 开始前，先修改 PRD，而不是直接建接口或写 service。
- PRD 至少补齐以下内容:
  - 产品目标
  - MVP 范围
  - 非目标
  - 输入
  - 输出
  - 页面交互
  - 算法口径
  - 验收标准
  - 性能约束
  - 已知问题

### 这一阶段必须回答的问题
- 这个 Part 到底解决什么问题
- 首版必须交付什么，明确不做什么
- 前端首屏需要什么，异步区块需要什么
- 返回字段的语义是否稳定
- 是否存在历史代码语义和当前 PRD 不一致的地方

## 2.2 第二步: 冻结 API 契约
- 在动手编码前，先冻结接口拆分和 schema。
- 需要先定下:
  - endpoint 列表
  - 请求参数
  - 响应字段
  - warning 结构
  - 分页规则
  - 默认值
  - 错误码语义

### 默认接口拆分原则
- `meta` 类接口: 首屏只读元信息
- `summary` 类接口: 核心指标摘要
- `list` 类接口: 分页列表
- `detail` 类接口: 单对象明细
- `compute` 类接口: 昂贵计算或带参数重算能力

### 禁止事项
- 不要用一个接口返回所有页面数据
- 不要让同一个字段在不同接口里语义不同
- 不要在前端联调阶段再临时改字段名

## 2.3 第三步: 做真实 dry run，验证关键假设
- 在实现前，先用至少 1 个真实数据集或 checkpoint 做 dry run。
- dry run 的目标不是完成业务，而是提前戳穿错误假设。

### dry run 必查项
- 数据输入 shape 是否符合预期
- 标签 shape 是否符合预期
- `TRAIN / TEST` 是否真的对应原始语义
- checkpoint 是否存在
- checkpoint 输出维度是否和配置一致
- 底层模块是否返回 PRD 所需字段
- 计算耗时是否大致可接受

### dry run 结束后必须产出
- 一份确认过的“真实口径”
- 一份“配置与现实的偏差清单”
- 一份“哪些字段可直接暴露，哪些需要二次加工”的结论

## 2.4 第四步: 先写 service，再写 router
- 推荐落地顺序:
  - `constants`
  - `schemas`
  - `services`
  - `api`
  - `app` 注册

### 分层约束
- `constants`: 版本号、默认阈值、支持范围、全局常量
- `schemas`: 纯响应/请求模型
- `services`: 真实业务逻辑、适配、缓存、聚合
- `api/router`: HTTP 参数校验、调用 service、输出 response_model
- `app`: 注册 router、挂调试页、健康检查

### 实施要求
- router 不直接操作底层模型和数据集
- service 不应直接耦合前端页面细节，但要支撑页面数据流
- schema 要为 warning、mapping、meta 预留位置

## 2.5 第五步: 把“已知不一致”产品化
- 发现偏差后，不要做隐式修补，而要显式建模。

### 常见偏差类型
- 配置文件和 checkpoint 不一致
- 数据命名和真实语义不一致
- PRD 保留字段但 MVP 暂不支持
- 不同数据集之间行为不一致
- 历史实现和当前产品设计不一致

### 标准处理方式
- PRD 补充 `MVP 备注 / 已知问题`
- API 返回 `warnings`
- 必要时在响应中增加映射字段
- 调试页或联调说明中同步记录

## 2.6 第六步: 按首屏优先顺序实现
- 每个 Part 默认按以下顺序推进:
  1. 首屏元信息
  2. 核心摘要结果
  3. 分页列表
  4. 单对象详情
  5. 参数化重算
  6. 调试页/联调页面

### 原因
- 这样可以先跑通最小闭环
- 可以尽早让前端接入
- 可以避免把计算压力集中到单一接口

## 2.7 第七步: 加缓存、分页和性能保护
- 只要涉及模型推理、聚类、归因、采样、批量统计，就优先设计缓存。
- 列表结果必须分页。
- 对可重复请求的昂贵结果，优先缓存中间结果而不是只缓存最终响应。

### 需要优先考虑缓存的内容
- 数据集加载结果
- 测试集批量预测结果
- 聚类结果
- 匹配分数 `I`
- players 构建结果
- what-if 评估结果
- shapley 采样结果

### 缓存设计要求
- 明确缓存键
- 缓存键必须覆盖影响结果的关键参数
- 明确缓存是否受 seed、threshold、cluster_k、ckpt 版本影响

## 2.8 第八步: 联调前做回归检查
- 至少检查以下三类情况:
  - 正常路径
  - 边界参数
  - 明确错误输入

### 回归检查清单
- 接口字段和 PRD/API 文档一致
- warning 是否按预期返回
- 分页是否正确
- 排序是否稳定
- 随机种子固定后结果是否复现
- 性能是否满足该 Part 的 MVP 验收目标

## 3. 从 Part A 沉淀出的具体规则

### 3.1 不信 yaml，先信 checkpoint
- `Part A` 中分类类别数不能直接信 yaml。
- 后续凡是类别数、shapelet 数量、窗口长度、输出维度等信息，都优先从 checkpoint 或运行结果推断。

### 3.2 不信数据命名，先验真实语义
- `Part A` 中 `TRAIN / TEST` 与底层原始 split 存在映射反转。
- 后续凡是 split、标签名、类别名、时间粒度等概念，都要经过实际数据验证。

### 3.3 不假设输入 shape 恒定
- 同一类数据结构在不同模块里可能出现不同 shape。
- 后续所有 Part 都要先做标准化适配，再进入核心计算。

### 3.4 不做单接口“大总包”
- `Part A` 已经证明，重型 `/overview` 不利于首屏、调试、缓存和前后端联调。
- 后续所有 Part 默认拆成多个细粒度接口。

### 3.5 不把限制能力伪装成已支持能力
- `dataset_file` 在 PRD 中可以保留，但 MVP 若不支持，必须明确说清楚。
- 后续所有保留项都按同样规则处理。

### 3.6 不把工程真相藏起来
- warning 不是可选项，而是产品化透明度的一部分。
- 后续任何算法口径偏差、配置偏差、历史兼容问题，都要体现在响应中。

## 4. 新 Part 启动模板

### 4.1 启动前 Checklist
- [ ] PRD 已补全该 Part 的目标、范围、算法口径、验收标准
- [ ] API 字段和 endpoint 已冻结
- [ ] 已完成至少 1 次真实 dry run
- [ ] 已识别已知问题和 warning 输出策略
- [ ] 已确认首屏数据流和异步数据流
- [ ] 已确定缓存键和随机种子策略
- [ ] 已明确哪些能力是 MVP 不做但未来保留

### 4.2 编码顺序 Checklist
- [ ] 新增或更新 `constants`
- [ ] 新增或更新 `schemas`
- [ ] 实现 `service`
- [ ] 实现 `router`
- [ ] 注册到 `app`
- [ ] 增加调试入口或最小联调方式

### 4.3 联调前 Checklist
- [ ] 正常请求可返回
- [ ] 边界参数可处理
- [ ] 错误参数有明确错误码
- [ ] warning 与 mapping 信息完整
- [ ] 分页可用
- [ ] 缓存生效
- [ ] 响应字段与文档一致

## 5. 推荐文档维护方式
- 每完成一个 Part，都补两类内容:
  - 本文件中新增一条“该 Part 新踩出的可复用规则”
  - 对应 PRD 中补充该 Part 的 `已知问题 / 决策记录`

### 建议规则
- 可复用规则写在这里
- 单个 Part 的细节和例外写回对应 PRD
- 如果某条规则已经被多个 Part 证明有效，再升级为全局规范

## 6. 一句话流程总结
- 先改 PRD并冻结口径
- 再做 dry run 验证真实世界
- 再定 API 契约和 warning 策略
- 再按 `constants -> schemas -> service -> api -> app` 落地
- 最后补缓存、分页、联调和回归检查
